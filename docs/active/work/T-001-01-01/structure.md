## Target manifest content (the blueprint)

`.flox/env/manifest.toml` after this ticket (flox generates the skeleton; the `[install]`
entries and `systems` widening are our edits, applied via `flox install` + `flox edit`):

```toml
version = 1

[install]
nodejs.pkg-path = "nodejs"
nodejs.version = "24.16.0"
just.pkg-path = "just"
just.version = "1.54.0"

[vars]
# (empty ‚Äî deliberately)

[hook]
# (empty ‚Äî deliberately)

[profile]
# (empty ‚Äî deliberately)

[options]
systems = ["aarch64-darwin", "x86_64-darwin", "aarch64-linux", "x86_64-linux"]
```

Exact generated-section syntax may differ cosmetically by flox version (comments, key order);
the load-bearing content is the two exact-versioned `[install]` entries and the four-system
`options.systems` list.

`manifest.lock` is entirely tool-owned: never hand-edited, always committed as regenerated.

`env.json` will carry the environment name ‚Äî flox derives it from the directory (`mahjong`);
accept the default.

## Interfaces / boundaries

- **Public interface of this ticket**: the command form `flox activate -- <cmd>` resolving
  `node` (24.16.0) and `just` (1.54.0) on PATH, from a fresh clone, on any of the four systems.
  This is the contract T-001-01-02's justfile recipes and every later session build on.
- **Boundary with npm**: flox pins the runtime layer (node, npm-bundled) and non-npm CLIs
  (just). Everything installable via `package.json` stays out of the manifest ‚Äî that boundary
  keeps the manifest stable while JS deps churn.
- **Boundary with lisa/vend**: none of `.lisa*`, `.vend`, `board.svg` are touched.

## Ordering of changes

Order matters only in that the flox CLI owns file generation:

1. `flox init` at repo root ‚Üí creates `.flox/` skeleton (`env.json`, `env/manifest.toml`,
   `.flox/.gitignore`).
2. Widen `options.systems` to the four-system list (via `flox edit -f` with the prepared
   manifest, or editing then letting flox re-lock). Doing this *before* installs means each
   subsequent install resolves for all systems in one pass.
3. `flox install nodejs@24.16.0 just@1.54.0` ‚Üí writes `[install]` entries and regenerates
   `manifest.lock` with per-system resolutions.
4. Verify activation (`flox activate -- node --version`, `-- just --version`).
5. Fresh-clone simulation: activate from a clean checkout of the working tree state.
6. Single commit of `.flox/` (plus work artifacts per repo convention).

Steps 2‚Äì3 may collapse into one `flox edit` applying the full target manifest; the plan phase
picks the concrete sequence, but the invariant is: **flox regenerates the lock; we never write
`manifest.lock` by hand.**
