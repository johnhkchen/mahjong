## Per-file contracts

### `justfile`
Five recipes. Public: `dev`, `test`, `check`, `build` — each depends on `_deps` and execs
`flox activate -- npm run <name>`. Private `_deps`: runs `flox activate -- npm ci` unless
`node_modules/.package-lock.json` exists and is newer than `package-lock.json` (one shell
conditional; `[private]` attribute so `just --list` shows only the four public verbs). No
`deploy` recipe (T-001-03-02). Recipes are the *only* place flox is invoked; npm scripts stay
flox-agnostic so they also work inside an already-activated shell.

### `package.json`
- `"private": true`, `"type": "module"` (Vite-native ESM; svelte.config.js can use `export
  default`).
- `scripts`: `dev` → `vite`, `test` → `vitest run`, `check` → `svelte-check --tsconfig
  ./tsconfig.json && tsc -p tsconfig.node.json --noEmit`, `build` → `vite build`.
- `devDependencies` (exact, per D3): svelte 5.56.4, vite 8.1.3, @sveltejs/vite-plugin-svelte
  7.1.2, vite-plugin-singlefile 2.3.3, vitest 4.1.9, svelte-check 4.7.1, typescript 5.9.3,
  @tsconfig/svelte (latest at install time, exact-pinned).
- No `dependencies` — svelte is a devDep; the compiler output Vite inlines has no runtime
  package to resolve at serve time.

### `index.html`
Minimal valid document: `lang="en"`, viewport meta, `<title>mahjong</title>`, `<div
id="app">`, `<script type="module" src="/src/app/main.ts">`. No CSS link (styles ride in the
component), no icons/manifest (out of scope until the PWA ticket).

### `vite.config.ts`
```
/// <reference types="vitest/config" />          // types the `test` key
defineConfig({
  plugins: [svelte(), viteSingleFile()],
  test: { environment: 'node', include: ['src/**/*.test.ts'] },
})
```
Singlefile's `useRecommendedBuildConfig` default handles inline-assets/no-chunking (D5); we do
not restate build options.

### `svelte.config.js`
`{ preprocess: vitePreprocess() }` from `@sveltejs/vite-plugin-svelte`. Exists as its own file
(not inlined into vite config) because svelte-check discovers it independently of Vite — this
is the file that lets `lang="ts"` components typecheck.

### `tsconfig.json` (src context)
Extends `@tsconfig/svelte/tsconfig.json`; adds `strict: true` explicitly, module resolution
`bundler`, `noEmit`-style checking (svelte-check drives it; tsc never emits from this config),
`include: ["src/**/*.ts", "src/**/*.svelte"]`. Covers core, app, and the tests.

### `tsconfig.node.json` (tooling context)
Standalone (no extends): `strict`, `module`/`moduleResolution` bundler-compatible ESNext,
`types: ["node"]` omitted (config uses no node APIs; add only when needed), `noEmit: true`,
`include: ["vite.config.ts"]`. Checked by the `tsc -p` half of `check`.

### `src/core/index.ts`
```ts
export const ENGINE_NAME = 'mahjong-core'
```
One honest constant. Zero imports of any kind — the purity invariant holds vacuously and
grep-verifiably. T-001-02-01 grows this into the tile domain (or replaces it with real
modules; no downstream contract binds to `ENGINE_NAME`).

### `src/core/index.test.ts`
Imports `{ ENGINE_NAME }` from `./index` and `{ describe/it/expect }` from `vitest`; asserts
the constant. Proves: include-glob matches, TS transform runs, core module resolution works —
the idiom (colocated `*.test.ts` beside the module under test) every engine test follows.

### `src/app/main.ts`
```ts
import { mount } from 'svelte'
import App from './App.svelte'
mount(App, { target: document.getElementById('app')! })
```
Svelte 5 `mount()` API (not the removed `new App()`). Exports the mount result as default per
template convention — harmless, aids HMR.

### `src/app/App.svelte`
`<main>` with one line of placeholder text identifying the scaffold, `<style>` block with a few
rules (so the build proves CSS inlining too). **No import from `src/core/`** (D6 rationale:
first core→app flow is T-001-03-01's acceptance boundary). No runes yet — nothing stateful to
model.

### `src/app/vite-env.d.ts`
`/// <reference types="vite/client" />`. Lives in `src/app/` (not `src/`) because only app code
may ever touch Vite client types; core must stay platform-free even at the types level.

## Boundaries established (the actual deliverable)

- `src/core/` — pure TS, zero imports today, tests colocated. The grep gate downstream tickets
  rely on (`no DOM/Svelte imports in src/core/`) passes by construction.
- `src/app/` — everything platform-flavored: the entry point, the component, the Vite ambient
  types. `index.html` at root is Vite's mandated location, pointing into `src/app/`.
- Import direction: nothing imports app; app doesn't import core *yet* (deliberate, see D6).

## Ordering constraints

1. `package.json` before `npm install` (generates the lockfile) before anything runs.
2. Config files (`vite.config.ts`, `svelte.config.js`, tsconfigs) before source, so each check
   can run the moment its inputs exist.
3. `src/` files, then `justfile`, then end-to-end verification of all four recipes — the
   justfile is last because it's the integration layer the AC exercises.
4. Fresh-clone verification (clone the commit into scratch, run all four recipes) after commit
   — the AC's own wording, and T-001-01-01 set the precedent.
