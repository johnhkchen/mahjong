## The new test file, internally

`src/app/table.tap.svelte.test.ts` — the name is load-bearing twice: matches
vitest's `src/**/*.test.ts` include, and carries the `.svelte.` infix that
makes vite-plugin-svelte transform its runes (`$state` props proxy).

```
/** @vitest-environment jsdom */
// header comment: the E-007 named-risk guard — what it pins and why client-side

imports:
  svelte            → mount, unmount, flushSync
  vitest            → describe/it/expect/afterEach (or per-mount try/finally)
  ../core           → foldRecord, legalActions, kindIndexOf, kindOf,
                      type HandAction, type TileId, type TableState
  ./drive           → PLAYER, tapDiscard, tapClaim, claimChoices
  ./Table.svelte    → the component under guard

helpers (file-local):
  sortedDisplay(hand): TileId[]
      the in-test replica of Table's display sort — stable copy-sort by
      kindIndexOf(kindOf(id)); THE contract statement, deliberately duplicated
      (design: a comparator change in the component must fail here visibly)
  mountTable(table): { taps: TileId[], props, target, handButtons(), drawnButton(), done() }
      creates target div in document.body, props = $state({ table, ontap }),
      ontap pushes into taps; mount(Table, { target, props }); done() unmounts
      and removes target. handButtons() queries
      '[aria-label="your hand"] button.tap' in document order — document
      order ≡ visual order per research's flex-row equivalence note.
  tsumogiriTurns(live, n)
      the same frozen script builder drive.test.ts and app.ssr.test.ts each
      carry locally (third local copy, matching the established pattern —
      the suites deliberately do not share fixture code)

fixtures (module level, folded once, mirroring drive.test.ts's anchors):
  SEED = 1
  afterEastDraw   = fold(draw seat 0)         — state A
  seed-15 records — the pon-claim reorder (state C), via tsumogiriTurns(…, 8)
                    + the claim element taken from legalActions via tapClaim

describe('sorted hand render order')
  it: fixture sanity — draw order ≠ sorted order; duplicate kinds present
  it: 13 buttons; aria-label sequence equals sortedDisplay kinds
describe('tap → exact TileId at every sorted position')
  it: click k ⇒ ontap receives sortedDisplay[k], all 13 positions
      (duplicate-kind positions included — ids, not kinds)
  it: each received id → tapDiscard returns the offered element (toBe),
      action.tile === sortedDisplay[k]
  it: drawn button reports exactly table.drawn; absent from the sorted list
describe('keyed reorder preserves the mapping — tedashi merge')
  it: capture per-id button nodes; append REAL tapDiscard(sorted[0]) element;
      refold; props.table = next; flushSync()
      ⇒ 13 buttons, document order equals new sortedDisplay
      ⇒ click every position: exact new ids (now the owner's no-op — still
        the component's mapping fact)
      ⇒ surviving ids: SAME node (===) as before, moved not repurposed
      ⇒ discarded id's node detached; merged (ex-drawn) id: a NEW node
describe('keyed reorder preserves the mapping — claim shrink (seed 15)')
  it: mount at the claim window, apply the pon (13→11, two ids leave, from a
      DIFFERENT part of the sort), flushSync ⇒ same order/click/node-identity
      assertion battery
```

Assertion vocabulary follows house style: `toBe` for identity with
`legalActions` elements and DOM nodes, `toEqual` for id sequences, fixture
sanity asserted inline (drive.test.ts's "mined, never regenerate" discipline —
except nothing here needs mining: seed 1 and seed 15 anchors are already
frozen by the sibling suites).

   suite can even collect.

   clause d), `just check` (svelte-check must accept the rune test file).
4. `just build` — belt-and-braces proof the devDep never touches the single
   file (byte-size gate unchanged).
