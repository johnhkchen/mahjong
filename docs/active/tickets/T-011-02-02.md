## Context

Consecutive claim windows must read as distinct events, not one stuck dialog: a fresh
prompt enters with a visible beat (CSS-only, ~150-250ms per E-007 conventions,
`prefers-reduced-motion` honored) and the prompt is keyed per window (claimable
seat+tile) so Svelte never patches one window's DOM into the next. No artificial delay
beyond the existing pacing — distinctness, not throttling.

_Advances: P4, P2_

## Acceptance Criteria

- [ ] The prompt is keyed on the window identity (a new claimable discard remounts it);
  the entry transition exists and is disabled under prefers-reduced-motion; the repro
  suite's consecutive-window sequence asserts remount (not patch) semantics; SSR output
  unchanged.
