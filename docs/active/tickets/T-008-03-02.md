## Context

The table keeps going: App's authoritative state becomes the GameRecord; "next hand" (on
the score screen) appends the ended hand and deals fresh from the derived seed with scores
carried and dealer/winds rotated by the game fold; "new game" resets to a fresh GameRecord
(absorbing the interim header button's seed redraw). All transitions through the drive
seam; the view never invents game state.

_Advances: P1, P2_

## Acceptance Criteria

- [ ] From a won or drawn hand in `just dev`, "next hand" starts the following hand with
  carried scores and correct dealer/wind rotation, and "new game" resets scores to 25000
  with a fresh random seed (the `?seed=` boot pin still reproducing a full game). Drive/SSR
  tests cover both controls; the existing new-game header behavior is absorbed without
  regression.
