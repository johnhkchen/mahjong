## Intent — the bigger-picture play

Diagnosed surface: the engine is sound (the fold closes `claimable` on every draw; one
prompt render site), but the drive/view interaction has two legibility defects. (1) A
player's chi/pon answer enters `settleWindow` arbitration and can LOSE to a bot's
higher-priority claim (pon over chi, ron over pon) — rules-correct atamahane/priority,
but the tap is discarded with zero feedback, and the winner's next discard often opens
a similar-looking window ~250ms later: the dialog "didn't take" and "appeared twice."
(2) The houtei `dismissed` flag outlived its hand after E-008 continuation — fixed
inline 2026-07-04 (commit 3bcf9d3) — the fix needs regression coverage here. This epic
makes every window resolution *visible*: an outcome notice when a call is outranked
(who took the window, with what), a distinct presentation beat between consecutive
windows so a new prompt never reads as the old one re-appearing, and an interaction
test harness that scripts these exact sequences.

## Done looks like

Scripted interaction tests reproduce the double-dialog sequence and pin the fixed
behavior: when the player's call loses a window, a transient console notice names the
winner and their claim before the next prompt appears; consecutive windows are visually
distinct (a fresh-prompt beat, never an identical instant re-render); the console
cascade (claim prompt > outcome notice > riichi prompt > tenpai hint) is documented and
tested; the houtei dismissal reset at hand boundaries has a regression test. All in
`src/app/` — core untouched; both terminologies covered; single-file gate green.

## Context & constraints

View/drive-only: `settleWindow`'s arbitration semantics are correct and stay; this epic
changes what the player *sees*, not what folds. No new dependencies; CSS-only motion
per E-007 conventions; `prefers-reduced-motion` honored. Out of scope: changing claim
priority rules, engine furiten/yaku gates (E-009 shipped their badges), sound, undo.
